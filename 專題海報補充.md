# 專題題目：An Efficient Winograd-Based CNN Accelerator on FPGA
<!-- v3 修正版：數據統一為最終 timing-closed bitstream（19.06 FPS 那組）；
     新增雙時脈 CDC、Early Release、PE 13 級；資源為 v6.4 routed 數字 -->

**硬體平台：** PYNQ-Z2 FPGA
**整體設計：** 針對 VGG-16 做硬體加速，並使用 INT8 量化來降低運算與記憶體成本。

**實驗結果 (Tiny ImageNet 測試集)：**

* **19.06 FPS**
* **47.76 GOPS**
* **166 MHz**
* 功耗 4.58 W（PS 1.526 + PL 3.053，Vivado 報告）→ **10.43 GOPS/W**
* 相較於 FP32 baseline，準確率**只下降 0.51%**（INT8 = 61.79%）

## 1. Software Processing & Scheduling（左欄上半）

### Step 1：Operator Folding (算子折疊)

> 「把 Conv 跟 BatchNorm 融合成單一 conv layer。BN 在訓練時是 `(x−μ)/σ × γ + β`，推論時這四個都是常數，可以直接折進 conv 的 weight 跟 bias。」

### Step 2：Offline Winograd Transform

> 「因為 weight 已知，所以能先 transform 好——硬體只看到 4×4 weight，runtime 不用做 weight transform，代價是傳的資料量變大。」

### Step 3：Hardware-Aware PTQ (Hardware-Aware Post-Training Quantization)

1.利用 500 張校正集抓出活化值範圍，並對權重進行 Per-Channel INT8 量化，讓每個輸出通道都有獨立的最佳比例尺。
2.累積器 (ACC) 算出的結果必須透過公式 `ACC * (S_next/S_in * S_w)` 進行再量化。將這個浮點數 Scale 近似為一組「整數乘數 (Multiplier)」與「位移量 (Shift)」，也就是 `ACC * mult >> shift`。
3.當 BN 融合後的 Bias 乘上S_w可能撐爆硬體位寬，演算法會主動壓縮該通道的權重比例，會導致少數通道死亡

### Step 4：Hardware-Aligned Packing

* **16-channel aligned：** 硬體 1 個 chunk 處理 16 個 output channel，weight 補成 16 倍數。
* **Asymmetric padding：** 特徵圖 (FM) 兩側補 1 圈 0 變 `(W+2)×(H+2)`，再向上補齊 4 倍數讓 Winograd tile 整齊。
* **Build layer template (C code)：**
> 「為了消除 Runtime 時 CPU 重組資料的延遲，在初始化時，就用 C code 把這些對齊好的 Weight、PARAM、以及 Padding 區塊，在 DDR 的 TX Buffer 中排布成一個一體化、連續的二進位樣板 (Layer Template)。在實際運行推論時，CPU 只需要將新輸入的圖片數據直接複寫到樣板預留的 Image 區塊中。」

## 2. Static Staggered Scheduling

**目的：** 減少 FPGA 等待 CPU pack 時間。A 比 B 領先一層，讓兩張 image 在不同 layer 上錯開——CPU 永遠在 pack/unpack，並減少 FPGA IDLE。

### 為什麼要 Unpack？

FPGA 算完一層後，吐出來的資料依然是為了 16-Channel 還有 Winograd 對齊硬補了一堆 0 的格式。CPU 必須把這些無效的 0 砍掉，才能拿到正確的 Output。

### 為什麼用 Staggered 也遮不了 Unpack 的延遲？

Pack 跟 Unpack 的差別在「buffer 何時 ready」：

* **Pack 可以遮：** CPU 寫 TX buffer，FPGA 之後讀。Buffer 只要在 launch_dma 前 ready 就好——所以 CPU pack(B) 可以跟 FPGA compute(A) 平行。
* **Unpack 沒法遮：** 是因為 CPU 和 FPGA 共用同一塊 RX Buffer。當 Output FIFO 排空，CPU 終於開始讀取 RX Buffer 進行 Unpack 時。因為 Buffer 正在被 CPU 佔用，FPGA 只能被迫 IDLE 了。

## 3. Overall Hardware Architecture（左欄下半）

CMD decoder 看每筆資料前面的 header，分發到 PARAM bank、WEIGHT bank、FM bank、Global Cache。所有 bank 都是 dual-bank ping-pong——bank 0 算的同時，bank 1 載下個 input channel。

### 雙時脈域架構（CDC）

* **加速器域 166 MHz（FCLK_CLK0）**：CNN pipeline 全部（PE 陣列、transform、FIFO、512→128 converter）＋ DMA 的 MM2S 通道。
* **DMA 域 143 MHz（FCLK_CLK1）**：DMA 的 S2MM 通道與 AXI-Lite 控制。
* 跨域由 AXI-Stream Clock Converter（S2MM 方向）與 DMA 內建 async CDC 處理。
* 設計理由：S2MM 內部的 24-bit BTT counter carry chain 是 166 MHz 唯一收不掉的 IP 路徑；把它放到 7 ns 域、而把吞吐主力的 MM2S（權重方向）留在 166 MHz，**throughput 零損失**地完成時序收斂。

### Bank 切換：前端釋放（Early Release）

非輸出 packet（同 chunk 的中間 input channel）切 bank 時只等「前端排空」（sliding window / tile FIFO 空）＋固定 guard，**不等 PE 13 級排空**——權重切換時序與 psum RAW 由 pipeline 深度差保證安全。VGG-16 每張圖約 10 萬次 bank 切換，每次省 ~7 cycles，貢獻約 4% throughput。

### 封包格式

DMA 用 AXI-Stream 128-bit。一個 layer 由多個 chunk 組成，每個 chunk 對每個 input channel 送一個 packet。每個 packet 分 4 段（每段都有自己的 opcode header）：

* **PARAM header + 6 beats**（16 PE × 40-bit param + padding）
* **WEIGHT header + 16 beats**（16 PE × 4×4 INT8 weight）
* **CMD header**（1 beat 本身就是控制訊號：acc_mode / relu_en / pool_en）
* **FM payload**

## 4. Hardware Architecture（右欄上半）

*(重點：Shared Input Transform + Sliding Window + Tile FIFO Toggle)*

Tile generate(Sliding window) 當蓄水期結束，同一 cycle 同時吐 2個 tile（左半 A + 右半 B），但下游 Input Transform 一拍只能處理 1 個。所以用 2 個 FIFO 同時接這 2 個 tile，下游 toggle 輪流讀，保持 B→A 順序、不丟資料。

### PE 內部 13 級 Pipeline

| Cycle (拍) | 控制訊號 (Valid) | 運算內容 (Operations) |
| --- | --- | --- |
| **1** | val_1 | 接收廣播的 V 並鎖存 |
| **2** | val_2 | Element-wise mul 前半：partial product（LUT 乘法拆 V×W[3:0] / V×W[7:4]；DSP 走 MREG） |
| **3** | val_3 | Element-wise mul 後半：移位合併（DSP 走 PREG） |
| **4** | val_4 | A^T reduction (Column reduction) |
| **5** | val_5 | Row reduction & 讀取 Psum BRAM |
| **6** | val_6 | 跨通道累加 (Accumulate) & 寫回 BRAM |
| **7** | val_7 | 加上 Bias 與 ReLU |
| **8** | val_8 | Requantization 乘法（DSP MREG） |
| **9** | val_9 | Requantization 乘法（DSP PREG） |
| **10** | val_10 | Requantization Shift（38-bit barrel shift） |
| **11** | val_11 | Overflow 偵測 + Clip 截斷 |
| **12** | val_12 | 第一層 MaxPool (L1 比較) |
| **13** | valid_out | 第二層 MaxPool (L2 比較) & 寫入 Output FIFO |

乘法與 requantization 各兩級全 pipeline（DSP MREG+PREG 全開）、shift 與 clip 分拍——全部單週期路徑短於 6 ns。Throughput 每拍一筆不變，latency +3 拍（由 Early Release 機制反賺回來）。

### 4-lane Output FIFO

> 「16 PE 同時吐 output 太寬，拆 4 個 lane FIFO 平行收（每 lane 收 4 PE）。」

## 5. Performance Analysis（右欄中半）

### Software Scheduling Strategy 表

> 「三段優化結果：Sequential 12.6 → Staggered 17.3 → +Cache 18.9 FPS（量測於 pre-closure build）；最終 timing-closed build 為 **19.06 FPS**，總共 **+51%** throughput。」
> （註：三段 sweep 是舊 bitstream 量的，相對結論不變；若時間允許，在最終 bitstream 重跑三組讓整條線一致。）

### VGG-16 Bandwidth Reduction 圖

> 「Cache 設計的效果：FM traffic 從 17,582 KB 降到 4,669 KB，省 73.4%。Total DMA 從 44.8 MB 降到 31.9 MB，省 28.8%。Weight 沒變因為 weight 不能 cache（每個 chunk 的 weight 不同）。」

### Layer-wise Bottleneck 折線圖 (Memory Wall 問題)

One-shot 整層的所有 chunk 預先打包成一段連續的 TX buffer，DMA 一筆 launch 全部送進去。
*藍線是 GOPS、紅線是 DMA bandwidth。*

* **FPGA 時間:** `launch_dma` 到 `wait_dma`
* `launch_dma`: CPU 對 DMA 控制器下達傳輸指令
* `wait_dma`: CPU 等待 DMA 回報傳輸完成

* **DMA_bytes:** 該次傳輸的總資料量（位元組）
* `DMA_bytes = weight + FM_payload + headers

* **GOPS** = `2 × MACs / FPGA_time`
* `MACs = out_h × out_w × in_ch × out_ch × 9` (standard 3×3 conv)

* **BW (MB/s)** = `DMA_bytes / FPGA_time`

**分析結論：**

* **L1 (GOPS 159、BW 662 MB/s)：** tile 多、chunk overhead 攤平，屬於 compute-bound。
* **L10-12 (GOPS 21、BW 1325 MB/s)：** 只有 4 個 tile，chunk reload overhead 吃掉計算時間，屬於 memory-bound。
* （逐層數據量測於 pre-closure build；瓶頸結構與相對結論在最終 build 不變。）

### VGG vs RepVGG(Opt) + Bottleneck 段

> 「VGG 後段卡 memory wall，所以我也測了 RepVGG(Opt)。RepVGG 的 channel-balanced 設計讓 deep layer 維持 16×16 而不是 4×4，PE 利用率回升。結果：20.74 FPS（**+8.8%** vs 19.06）、56.67 GOPS（**+19%**），accuracy drop 2.57%（相對其自身 FP32 baseline；量化未特殊處理，部分 scale 被壓成 0）。」
> （註：RepVGG 數據若是舊 bitstream 量的，最終版要重跑一次；百分比一律以 19.06 為分母重算。）
